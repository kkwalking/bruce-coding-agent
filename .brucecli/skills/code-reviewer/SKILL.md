# Code Reviewer

执行高信号代码审查，优先寻找会影响真实运行结果的问题，而不是罗列风格偏好。

## 审查流程

1. 读取 `references/checklist.md`，使用其中的检查清单。
2. 确认审查范围：
   - 优先检查当前 Git 工作区的 staged、unstaged 和 untracked 变更。
   - 用户指定 commit、分支、PR 或文件时，以用户指定范围为准。
3. 阅读变更上下文和相关调用方，不要只看孤立 diff。
4. 检查实现是否符合仓库约定、现有接口和数据流。
5. 按风险选择并运行只读验证，例如编译、测试或静态检查；不要运行会重写文件的命令。
6. 只报告能够说明触发条件和实际影响的问题。

## 输出要求

先输出 findings，按严重程度从高到低排列：

- `[P0]`：立即造成严重破坏、数据损失或安全事故。
- `[P1]`：高概率导致核心功能错误或重大回归。
- `[P2]`：在合理场景下导致功能、可靠性或维护问题。
- `[P3]`：低风险但值得修复的问题。

每条 finding 必须包含：

- 简短标题。
- 精确文件和行号。
- 触发问题的条件。
- 对用户或系统的具体影响。
- 简洁且可执行的修复方向。

如果没有发现问题，明确写“未发现可操作问题”，并补充尚未验证的风险或测试缺口。

## 边界

- 默认只审查和报告，不修改代码。
- 不把纯格式、命名偏好或主观重构建议当作缺陷。
- 不重复报告同一根因。
- 不报告变更之外早已存在且与本次修改无关的问题。
- 不因测试通过就假设实现正确；同时也不要在缺乏证据时猜测故障。
